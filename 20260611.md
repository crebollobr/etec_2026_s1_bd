## 1. Por que dividir em várias tabelas

Imagine guardar cliente **e** pedidos numa tabela só: o nome do cliente se repetiria em cada pedido, gastando espaço e abrindo brecha para erros (o mesmo cliente escrito de duas formas).

A solução relacional é **separar** em tabelas e **ligá-las**:

- `cliente` — dados de quem compra
- `pedido` — cada compra, apontando para o cliente que a fez

Essa ligação é feita pela **chave estrangeira**.

---

## 2. Chave estrangeira (`FOREIGN KEY`)

Uma **chave estrangeira** é uma coluna que aponta para a **chave primária** de outra tabela. Ela garante a **integridade referencial**: não dá para cadastrar um pedido para um cliente que não existe.

### Montando o ambiente

```sql
CREATE DATABASE nome_sobrenome_20260611;
USE nome_sobrenome_20260611;
```

Tabela "pai" (lado do `1`):

```sql
CREATE TABLE cliente (
    id     INT AUTO_INCREMENT PRIMARY KEY,
    nome   VARCHAR(100) NOT NULL,
    cidade VARCHAR(60)
);
```

Tabela "filha" (lado do `N`) — repare na última linha:

```sql
CREATE TABLE pedido (
    id         INT AUTO_INCREMENT PRIMARY KEY,
    cliente_id INT,
    produto    VARCHAR(100) NOT NULL,
    valor      DECIMAL(10,2),
    FOREIGN KEY (cliente_id) REFERENCES cliente(id)
);
```

> `FOREIGN KEY (cliente_id) REFERENCES cliente(id)` lê-se: "a coluna `cliente_id` só aceita valores que existam em `cliente.id`" (ou `NULL`).

### A integridade na prática

```sql
INSERT INTO cliente (nome, cidade) VALUES ('Ana Silva',   'São Paulo');
INSERT INTO cliente (nome, cidade) VALUES ('Bruno Souza', 'Rio de Janeiro');
INSERT INTO cliente (nome, cidade) VALUES ('Carla Lima',  'Curitiba');
```

```sql
INSERT INTO pedido (cliente_id, produto, valor) VALUES (1, 'Teclado', 150.00);
INSERT INTO pedido (cliente_id, produto, valor) VALUES (1, 'Mouse',    80.00);
INSERT INTO pedido (cliente_id, produto, valor) VALUES (2, 'Cadeira', 600.00);
INSERT INTO pedido (cliente_id, produto, valor) VALUES (NULL, 'Caneca', 35.00); -- pedido sem cliente definido
```

```sql
-- Isto FALHA: não existe cliente com id = 99
INSERT INTO pedido (cliente_id, produto, valor) VALUES (99, 'Monitor', 900.00);
```

> ⚠️ O MySQL recusa o último comando com erro de *foreign key constraint*. É exatamente esse o objetivo da chave estrangeira: impedir dado inconsistente.

Repare nos dados que montamos de propósito:
- **Carla** não tem nenhum pedido.
- O pedido **"Caneca"** não tem cliente (`cliente_id = NULL`).

Esses dois casos vão fazer diferença nos joins a seguir.

## 3. Joins — combinando tabelas

`JOIN` junta linhas de duas tabelas usando uma condição de ligação (em geral `pedido.cliente_id = cliente.id`).

### 3.1 `INNER JOIN` — só o que casa nos dois lados

Retorna apenas as linhas em que **existe correspondência** nas duas tabelas.

```sql
SELECT  c.nome, c.cidade, p.produto, p.valor
FROM    cliente c
INNER JOIN pedido p ON p.cliente_id = c.id;
```

Resultado: **3 linhas** (Ana×2, Bruno×1).
Carla some (não tem pedido) e a Caneca some (não tem cliente).

> `c` e `p` são **apelidos** (*alias*) das tabelas — encurtam a escrita.

### 3.2 `LEFT JOIN` — tudo da tabela da esquerda

Traz **todas** as linhas da tabela à esquerda; quando não há par à direita, as colunas da direita vêm como `NULL`.

```sql
SELECT  c.nome, p.produto, p.valor
FROM    cliente c
LEFT JOIN pedido p ON p.cliente_id = c.id;
```

Resultado: aparece a **Carla** com `produto` e `valor` em `NULL`.
Ótimo para responder *"quais clientes nunca compraram?"*:

```sql
SELECT  c.nome
FROM    cliente c
LEFT JOIN pedido p ON p.cliente_id = c.id
WHERE   p.id IS NULL;
```

### 3.3 `RIGHT JOIN` — tudo da tabela da direita

O espelho do `LEFT`: traz **todas** as linhas da tabela à direita.

```sql
SELECT  c.nome, p.produto, p.valor
FROM    cliente c
RIGHT JOIN pedido p ON p.cliente_id = c.id;
```

Resultado: aparece a **Caneca** com `nome` em `NULL` (pedido sem cliente).

> 💡 Todo `RIGHT JOIN` pode ser reescrito como `LEFT JOIN` invertendo a ordem das tabelas. Na prática, `LEFT JOIN` é muito mais usado.

### Resumo visual

| Join | Mantém linhas sem par? |
|------|------------------------|
| `INNER JOIN` | Não — só o que casa nos dois lados |
| `LEFT JOIN`  | Sim, da **esquerda** (clientes sem pedido) |
| `RIGHT JOIN` | Sim, da **direita** (pedidos sem cliente) |

## 5. Resumo

| Comando | Para que serve |
|---------|----------------|
| `FOREIGN KEY ... REFERENCES` | Liga uma tabela a outra e garante integridade |
| `INNER JOIN` | Só linhas com par nos dois lados |
| `LEFT JOIN` | Tudo da esquerda + pares da direita |
| `RIGHT JOIN` | Tudo da direita + pares da esquerda |
| `ON` | Define a condição de ligação |
| alias (`cliente c`) | Apelido para encurtar a consulta |

**Boas práticas:**
1. A chave estrangeira sempre aponta para uma **chave primária** (ou coluna `UNIQUE`).
2. Ao apagar tabelas relacionadas, comece pela **filha**.
3. Prefira `LEFT JOIN` a `RIGHT JOIN` — mais legível.
4. Use **alias** para deixar o `JOIN` curto e claro.
